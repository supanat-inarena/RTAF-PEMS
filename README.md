# RTAF PEMS (Pilot Equipment Management System)
### ระบบจัดการอุปกรณ์และเวชภัณฑ์นักบิน กองทัพอากาศไทย (🔒 RTAF Secure)

RTAF PEMS เป็นระบบเว็บแอปพลิเคชันแบบ Full-stack ที่ออกแบบมาเพื่อการบริหารจัดการ ติดตามสถานะ และวางแผนการเบิกจ่ายอุปกรณ์เทคโนโลยีช่วยบินและอุปกรณ์ป้องกันภัยส่วนบุคคลสำหรับกำลังพลนักบินกองทัพอากาศ (เช่น G-Suit, หมวกบิน Helmet, หน้ากากออกซิเจน Oxygen Mask, เสื้อแจ็กเก็ตบิน Flight Jacket, และรองเท้าบูทข้าราชการบิน Boots) ระบบนี้ทำงานภายใต้มาตรฐานสถาปัตยกรรมความมั่นคงปลอดภัยกองทัพอากาศไทยเชื่อมต่อข้อมูลผู้ใช้งานกลางและทำงานอย่างเต็มรูปแบบร่วมกับฐานข้อมูลระบบคลาวด์ที่ทนทาน

## 1. เทคโนโลยีและภาษาที่ใช้ (Tech Stack)

ระบบนี้พัฒนาขึ้นด้วยโครงสร้างระดับอุตสาหกรรม (Enterprise-grade) เน้นความรวดเร็ว ปลอดภัย และบำรุงรักษาง่าย โดยใช้งานเครื่องมือดังต่อไปนี้:

*   **ภาษาหลัก (Programming Language):** 
    *   **TypeScript (TSX):** เพิ่ม Type Safety และลดโอกาสเกิดข้อผิดพลาดในการพัฒนาผ่านระบบ Interface และ Static Typing
*   **โครงสร้างส่วนหน้า (Frontend Framework):**
    *   **React 18 + Vite:** ช่วยให้อัตราการเรนเดอร์ UI รวดเร็ว และรองรับระบบคอมโพเนนต์แยกส่วนที่มีประสิทธิภาพสูง
    *   **Tailwind CSS:** สำหรับจัดการการแสดงผลที่ตอบสนองต่อผู้ใช้บนทุกขนาดหน้าจอ (Responsive Design) ด้วยรูปแบบ Utility-First
    *   **Lucide React:** ชุดไอคอนเวกเตอร์ที่คมชัดและเป็นระบบมาตรฐานเดียวกันทั้งหมด
    *   **Framer Motion:** ระบบแอนิเมชันสร้างการเคลื่อนไหวที่นุ่มนวลและเป็นระเบียบในการเปลี่ยนผ่านหน้าจอ
*   **โครงสร้างส่วนหลังและฐานข้อมูล (Backend & Cloud Database):**
    *   **Google Firebase Firestore:** ฐานข้อมูล NoSQL แบบเรียลไทม์ที่ทนทานสูง (Durable Cloud Persistence) เพื่อเก็บสถานะพัสดุและรายการเบิกจ่ายแบบปลอดภัย
    *   **Firebase Authentication:** ใช้ประมวลผลการจดจำผู้ใช้และรักษาความปลอดภัยของสิทธิ์ในการเชื่อมต่อข้อมูลเซสชัน

## 2. สถาปัตยกรรมและการทำงานแต่ละขั้นตอน (System Workflows)

กระบวนการดำเนินงานของ RTAF PEMS แบ่งออกเป็น **5 ขั้นตอนหลัก** ดังต่อไปนี้:

```
[ผู้ใช้กรอก Username ทอ.] ──► [รับรหัส OTP 6 หลัก] ──► [ตรวจสอบสิทธิ์ RTAF MFA]
                                                               │
                                                               ▼
[แดชบอร์ดภาพรวมผู้บริหาร] ◄─── [กรองเข้าสู่ระบบตามสิทธิ์] ◄─── [ประมวลผลข้อมูลสำเร็จ]
     │                      (Pilot / Supply / Commander / Executive / Admin)
     │
     ├─► นักบิน (Pilot): ยื่นขอรับพัสดุใหม่ (Create Requests)
     ├─► เจ้าหน้าที่พัสดุ (Supply): ตรวจพัสดุในคลัง (Inventory Control) & ดำเนินการจัดส่ง
     └─► ผู้บังคับบัญชา (Commander): อนุมัติ/ปฏิเสธคำร้องขอ (Approve/Reject Requests)
```

### ขั้นตอนที่ 1: การพิสูจน์ตัวตนผ่านระบบ RTAF MFA OTP 
*   **คำอธิบาย:** ระบบยกเลิกระบบบัญชีและรหัสผ่านแบบเก่าทั้งหมด เพื่อป้องกันการคาดเดารหัสผ่านและการรั่วไหลของข้อมูล โดยระบบจะใช้สถาปัตยกรรม **RTAF OTP (One-Time Password)**
*   **ข้อมูลนำเข้า:** ใช้บัญชี RTAF Email Prefix (เช่น `ukit_wu` โดยไม่ต้องใส่ `@rtaf.mi.th`) คู่กับรหัสผ่านแบบใช้ครั้งเดียวจำนวน 6 หลักที่สร้างจาก OTP App หรือ Line OA กลาง ทอ.
*   **การประมวลผล:** ระบบจะจำลองข้อมูลสแต็กและติดต่อฐานข้อมูลความมั่นคงปลอดภัยกลาง ตรวจสอบว่า `login_mode` อยู่ภายใต้สัญญารับสิทธิ์ `SWC-AUTH-Login` และคืนสิทธิ์ของบุคคลตามรายละเอียดในฐานข้อมูลกำลังพล

### ขั้นตอนที่ 2: การคัดกรองบทบาทเข้าใช้ระบบ (Role Routing)
*   **คำอธิบาย:** เมื่อยืนยันตัวตนสำเร็จ ระบบจะแมปประเภทกำลังพลและบังคับให้เข้าสิทธิ์แดชบอร์ดที่ผู้ใช้มีสิทธิ์เข้าถึงเท่านั้น โดยจะไม่แสดงส่วนนำทางที่ไม่เกี่ยวข้อง
*   **บทบาทที่รองรับ:** นักบิน (Pilot), นายทหารส่งกำลังพัสดุ (Supply Officer), ผู้บังคับการกองบิน (Commander), ผู้บริหารระดับสูง ทอ. (Executive), และผู้ดูแลระบบคอมพิวเตอร์ (Administrator)

### ขั้นตอนที่ 3: กระบวนการส่งคำร้องขอพัสดุของนักบิน (Pilot Request Flow)
*   **คำอธิบาย:** นักบินที่อุปกรณ์ชำรุดหรือครบกำหนดการเปลี่ยน สามารถเปิดหน้าฟอร์มคำขอเบิกอุปกรณ์ ระบุประเภทของพัสดุ จำนวนที่จำเป็น และแนบเหตุผลความจำเป็น
*   **การประมวลผล:** ข้อมูลจะถูกเขียนลงใน Google Firestore คอลเลกชัน `/requests` ทันทีในสถานะ `Pending (รอการตรวจสอบ)` พร้อมระบุประทับเวลา (Timestamp) ปัจจุบันอย่างแม่นยำ

### ขั้นตอนที่ 4: การตรวจสอบงบประมาณและอนุมัติโดยผู้บังคับบัญชา (Commander Approval Flow)
*   **คำอธิบาย:** ผู้บังคับการกองบินจะเห็นประวัติคำขอเบิกจากนักบินใต้สังกัดแบบเรียลไทม์ พร้อมตัวแสดงผลทางสถิติว่างบประมาณการเบิกในหน่วยงานและค่าเฉลี่ยความปลอดภัยของทีมอยู่ในระดับใด
*   **การประมวลผล:** ผู้บัญชาการสามารถกด **Approve (อนุมัติ)** หรือ **Reject (ปฏิเสธ)** หากได้รับการอนุมัติ สถานะของคำขอในคลาวด์จะเปลี่ยนเป็น `Approved` และเปิดสิทธิ์ส่งต่อนายทหารพัสดุ

### ขั้นตอนที่ 5: การส่งมอบและอัปเดตคลังพัสดุ ทอ. (Inventory Management Flow)
*   **คำอธิบาย:** นายทหารพัสดุสามารถเข้ามาดูรายการคำขอเบิกที่ผ่านการอนุมัติแล้ว และดำเนินการเตรียมพัสดุเพื่อจัดส่งให้แก่ข้าราชการนักบิน
*   **การประมวลผล:** เมื่อเจ้าหน้าที่ทำการอัปเดตสถานะเป็น `Dispatched` หรือจัดส่งสำเร็จ ระบบจะทำการหักลบจำนวนสินค้าคงเหลือ (Inventory Stock) ในคลังสินค้าส่วนกลางของ ทอ. โดยอัตโนมัติ

```
│       └── ExecutiveDashboard.tsx# หน้าจอนำเสนอตัวชี้วัดความพร้อมทางยุทธวิธีของกองทัพอากาศ
```

## 5. คู่มือการติดตั้งและเริ่มใช้งาน (Installation & Setup)

ทำตามขั้นตอนต่อไปนี้เพื่อเปิดรันแอปพลิเคชันบนคอมพิวเตอร์ของคุณ:

### 1. ความต้องการของระบบพื้นฐาน (Prerequisites)
*   **Node.js:** เวอร์ชัน 18.0.0 หรือสูงกว่า
*   **NPM:** เวอร์ชัน 9.0.0 หรือสูงกว่า

### 2. การดาวน์โหลดและติดตั้งไลบรารีที่จำเป็น
เปิดโปรแกรม Command Line หรือ Terminal และสั่งงานตามขั้นตอน:

```bash
# 1. ติดตั้งไลบรารีพึ่งพิงทั้งหมดตามที่ระบุใน package.json
npm install
```

### 3. การเชื่อมต่อฐานข้อมูล Google Firebase
สร้างไฟล์ `.env` ที่โฟลเดอร์นอกสุด และระบุค่าพารามิเตอร์คีย์การเข้าถึง (ไม่แนะนำให้นำค่าจริงอัปโหลดขึ้น Git สาธารณะ):
```env
VITE_FIREBASE_API_KEY=YOUR_API_KEY
VITE_FIREBASE_AUTH_DOMAIN=YOUR_AUTH_DOMAIN
VITE_FIREBASE_PROJECT_ID=YOUR_PROJECT_ID
VITE_FIREBASE_STORAGE_BUCKET=YOUR_STORAGE_BUCKET
VITE_FIREBASE_MESSAGING_SENDER_ID=YOUR_MESSAGING_SENDER_ID
VITE_FIREBASE_APP_ID=YOUR_APP_ID
```
*(ในกรณีทดสอบผ่านเครื่องมือ AI Studio คลาวด์พารามิเตอร์จะจำลองให้อย่างปลอดภัยโดยอัตโนมัติ)*

### 4. การรันระบบในโหมดการพัฒนา (Development Mode)
ใช้สคริปต์ด้านล่างนี้เพื่อรันระบบจำลอง และทดสอบเว็บเซิร์ฟเวอร์บนพอร์ตภายใน:
```bash
# รันเซิร์ฟเวอร์จำลองเพื่อทดสอบ (ใช้พอร์ต 3000 เป็นมาตรฐานหลัก)
npm run dev
```
หลังจากรันสำเร็จ สามารถเข้าชมหน้าเว็บได้ทาง URL: `http://localhost:3000`

### 5. การตรวจสอบความเรียบร้อยของโค้ด (Lint & Type Check)
สำหรับตรวจสอบข้อผิดพลาดด้านไวยากรณ์ (Syntax) และสกีมาคีย์ชนิดข้อมูลของโปรแกรม:
```bash
npm run lint
```

### 6. การคอมไพล์เพื่อนำไปใช้งานจริง (Production Build)
เมื่อระบบพร้อมสำหรับส่งมอบหรือรันบนคลาวด์ผลิตจริง ให้ทำตามสคริปต์นี้เพื่อรับไฟล์มินิฟายที่มีประสิทธิภาพสูงสุด:
```bash
npm run build
```
ไฟล์ผลลัพธ์จะถูกรวบรวมอย่างเป็นระบบระเบียบในไดเรกทอรี `/dist` เพื่อการส่งมอบที่ปลอดภัยและรวดเร็ว

## 6. มาตรการความมั่นคงปลอดภัย (Security Standards)

1.  **MFA Standard Only:** ระบบถอนสิทธิความลื่นไหลจากการใช้รหัสผ่านแบบเก่า และยืนยันตัวตนด้วย Multi-Factor Authentication (OTP 6 หลัก) เป็นเกณฑ์บังคับ 100%
2.  **Explicit Database Authorization:** กำหนดนโยบายเข้าถึงในฐานข้อมูล (Firestore Security Rules) ไม่ให้อ่านข้ามสิทธิ์ เช่น นักบินทั่วไปจะไม่สามารถเห็นหรือเข้าถึงระบบปรับเพิ่มจำนวนพัสดุของเจ้าหน้าที่พัสดุได้
3.  **No Core Secret Leaking:** ระบบความลับทั้งหมดและโทเค็น API คีย์เชื่อมต่อ จะถูกเข้ารหัสและเรียกทำงานผ่านฝั่งเซิร์ฟเวอร์ความเร็วสูงอย่างแน่นหนา ปลอดภัยจากภายนอก
